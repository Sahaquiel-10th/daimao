# 呆猫上线前检查与性能优化记录

更新时间：2026-07-01

## 已处理

### 小程序主包图片

原来 `miniprogram/images/daimao2` 内有 19 张 PNG，合计约 28MB，且多个页面直接引用这些本地图片。小程序主包因此过大，会影响上传、下载、首次打开和部分页面渲染。

本次处理：

- 所有 `/images/daimao2/...` 引用已迁移到 `miniprogram/config/assets.js` 的 CloudBase 云存储 fileID。
- `miniprogram/utils/assets.js` 增加 CloudBase 临时 URL 缓存，缓存有效期 12 小时。
- `miniprogram/images/daimao2` 已从主包移除。
- 小程序目录体积从约 28MB 降到约 528KB。

### 调试和种子入口

上线后不应默认开放测试/调试/种子数据入口。本次处理：

- `daimaoBusiness` 的 `debugEcho`、`testAiConnection`、`seedDemoData`、`adminDebug*`、`adminTest*` 默认关闭。
- `daimaoTagFunctions` 的 `debugMyConnections`、`debugConnectionWithUser`、`debugCurrentProfile`、`migrateProfilesToSql` 默认关闭。
- 如需临时排查，在对应云函数环境变量增加：

```text
ENABLE_DEBUG_ACTIONS=true
```

正式环境不要配置这个变量。

### 云函数上传忽略

为 `daimaoBusiness` 和 `daimaoTagFunctions` 增加 `.cloudignore`，尽量排除依赖中的 map、test、docs、examples、日志和 `.DS_Store`。

## 图片存储原则

UI 图片、头像、社区徽章、项目封面、活动封面都应该放 CloudBase 云存储或 COS，不要放 ReAI/向量库。

向量库只放文本证据链，用于 RAG 检索。图片不是证据检索对象，不进向量库。

## 仍然可能慢的地方

### 1. 云函数冷启动

`daimaoBusiness` 依赖较重，主要因为包含：

- CloudBase RDB SDK
- AI/RAG 请求逻辑
- `mammoth` 用于 docx 文本解析
- `pdf-parse` 用于 pdf 文本解析

这是当前功能需要，不是临时代码。但如果上线后仍明显慢，建议拆分：

- `daimaoBusiness`：保留核心业务读写。
- `daimaoEvidenceParser`：负责 docx/pdf/txt/md 解析。
- `daimaoRagIndexer`：负责 RAG 索引和重试。
- `daimaoAiReview`：负责 AI 申请初审。

这样主业务云函数会更轻，冷启动更快。

### 2. 图片临时 URL 获取

CloudBase fileID 转临时 URL 本身需要一次云函数/云 API 调用。当前已做 12 小时本地缓存。头像、社区徽章、封面如果仍慢，后续可继续做：

- 列表首屏只加载必要图片。
- `image` 增加 `lazy-load`。
- 后台上传时自动压缩成 WebP 或小尺寸 PNG。
- 数据库里保存封面缩略图 URL 和原图 URL 两份。

### 3. 数据列表一次拉太多

管理后台如果一次性拉用户、社区、项目、活动、证据链、图片临时 URL，会天然变慢。上线后用户多起来，应改成分页和按需加载：

- 列表页只拉摘要。
- 点详情再拉详情。
- 图片临时 URL 只给当前页生成。

## 不建议删除的内容

- `scripts`：当前包含 NFC 批量生成等业务脚本，按用户要求不处理。
- `呆猫视觉`：源视觉资产，按用户要求不处理。
- `miniprogram/utils/*MockStore.js`：目前由 `apiMode` 控制，正式配置是 `cloud`，不会在生产路径使用；可作为本地应急 fallback 暂时保留。
- `admin-web/src/api.js` 内 mock 数据：仅 `VITE_ADMIN_USE_MOCK=true` 时启用，正式部署不应配置该变量。
